- Конвенція версіонування
    - Надає чітку git-історію
    - Дає змогу реалізувати CHANGELOG.md
    - Зрозуміле виконання Rollback
- Це не лічильник коммітів (змін), а повний реліз до якого відповідна версія додається
- Структура версії: `X.Y.Z` = `MAJOR.MINOR.PATCH`
    - Невід'ємні цілі числа без ведучих нулів
    - Типи
        - PATCH: Виправлення
            - Зміна, після якої система поводиться так як мала поводитись спочатку
            - Користувач нічого нового не отримав
            - Приклад: `1.0.0` -> `1.0.1` -> `1.0.2` -> `1.0.3`
                - По одному багу на кожен реліз або 1.0.1, де одразу в одному релізі декілька виправлено
        - MINOR: Нова функціональність, якої не було раніше
            - Присутня зворотна сумісність:  все що було до працює і після
            - Користувач отримує щось нове поверх старого
            - Приклад
                - `1.0.3` -> `1.1.0`: додали темну тему
                    - ! При збільшенні, `PATCH` скидається до 0
                - `1.1.0` -> `1.2.0`: додали експорт
                - `1.2.0` -> `1.3.0`: додали фільтри
                - = Якщо щось було видалено таке, що не повпливало на клієнта, то `PATCH` або `MINOR`
                    - Видалено зайвий внутрішній модуль при рефакторингу
        - MAJOR: Щось змінилось так, що старий спосіб більше не працює
            - Приклади
                - Бекенд змінив структуру відповіді /api/auth
                    - Клієнт чекав одного на фронті, а по факту вже інше
                - Було видалено endpoint або функцію
                - Прибрали SSO логін
                - Змінилася авторизація
                - Змінили стейт-менеджер з Redux на Zustand з новою структурою
            - ! При збільшенні, `PATCH` і `MINOR` скидається до 0

### Особливості під час початкової розробки
- MAJOR = 0
    - Вcе може змінюватися будь-коли
    - ! Можна робити breaking changes в `MINOR` і це не порушує SemVer
- Коли `1.0.0`, то дається обіцянка про те, що API стабільний і SemVer повинен бути строгим`

### Pre-release
- Спосіб позначення, що версія не є стабільною
    - Проміжний стан перед фінальним релізом
    - Має нижче пріоритет, ніж стабільна
- Позначається дефісом після `PATCH` і серією ідентифікаторів через крапку
    - Числові ідентифікатори не можуть мати ведучих нулів
- Етапи (просто загально прийняті)
    - `alpha`: Рання розробка, де багато багів і все сильно міняється
    - `beta`: Основний функціонал готовий, але є баги
    - `rc`: Готовий до релізу, якщо не будуть знайдені критичні баги
    - Будь-які довільні: `canary`, `nightly`, `preview`, ...
- Приклади
    - `1.0.0-alpha`
    - `0.51.0-alpha.1`
        - & На практиці pre-release використовують перед значними релізами
    - `1.0.0-beta.11`
- Реальний сценарій для числового ідентифікатора
    - `1.0.0-alpha.1`: виклали на тестовий сервер, показали замовнику
    - `1.0.0-alpha.2`: виправили форму і рейтинг
    - `1.0.0-beta.1`: відкрили доступ більшій кількості користувачів
    - `1.0.0-beta.2`: оптимізували завантаження
    - `1.0.0-rc.1`: фінальна перевірка перед релізом
    - `1.0.0-rc.2`: виправили баг
    - `1.0.0`: стабільний реліз

### Build metadata
- Технічна інформація про конкретну збірку
    - Не впливає на пріоритет версій
- Позначається знаком `+` після `PATCH` або pre-release
- & На практиці у версіях продукта не використовується
- Приклади
    - `1.0.0+20240315143000`: Дата і час збірки
    - `1.0.0-beta+exp.sha.5114f85`: SHA коміту
    - `1.0.0+build.1337`: Номер збірки
